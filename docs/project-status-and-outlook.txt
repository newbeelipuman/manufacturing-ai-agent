制造业 AI Agent 后端 MVP 现场整理与后续展望

整理口吻：Codex 现场观察

一、当前项目真实状态

这个项目目前已经不是空架子，但也还不是一个厚实的 Agent 平台。

它更准确的定位是：一个面向制造业 ERP/MES/WMS 业务查询场景的后端 MVP，已经跑通了 P0 演示链路，能证明项目方向、模块边界和基本工程组织是成立的。

当前已经具备：

1. FastAPI 后端基础应用。
2. GET /health 健康检查。
3. SQLAlchemy 数据模型。
4. PostgreSQL Docker 演示环境。
5. SQLite 本地开发和测试 fallback。
6. 模拟销售订单、库存、工单、采购、知识库、审计日志、用量统计数据。
7. 5 个只读工具：
   - query_order_status
   - query_inventory_by_sku
   - query_work_order
   - query_purchase_arrival
   - query_exception_sop
8. /api/chat 自然语言入口。
9. 基础 Agent 路由。
10. 角色权限检查。
11. 工具调用日志 tool_call_log。
12. Agent 调用日志 agent_call_log。
13. usage 统计。
14. 基础 SOP 检索。
15. Docker Compose 启动配置。
16. README 和项目边界说明。
17. 回归测试，当前 pytest 通过。

我实际验证过：

- python -m compileall app 通过。
- pytest -q 通过。
- docker compose config 通过。
- 5 个 required demo questions 能通过 /api/chat 返回结果。

所以，当前项目是一个可演示的后端 MVP，而不是一个只停留在文档里的想法。


二、当前项目的短板

当前最大问题不是“完全没做”，而是“做得太像 demo”。

尤其是 Agent 编排层，目前还比较薄。

主要短板如下：

1. Agent 路由偏硬编码

当前 /api/chat 主要依靠关键词和固定编号判断问题类型，例如 O1001、WO1001、PO1001、SKU-KB-001。

这能支撑演示，但不能体现真正的泛化能力。

后续需要从：

关键词分支

升级为：

intent classification + entity extraction + tool planning + execution trace + answer composition


2. 实体抽取能力不足

现在更像是围绕 5 个固定问题做了适配。

如果用户换成其他订单号、其他 SKU、其他工单，系统理论上有数据库查询能力，但 Agent 层并没有真正稳定地从自然语言里抽取这些实体。

后续应支持：

- 订单号抽取
- 工单号抽取
- 采购单号抽取
- SKU 抽取
- 批次号抽取
- 多实体问题识别
- 缺实体时返回澄清问题


3. RAG 还是简单关键词检索

当前 query_exception_sop 更接近关键词匹配，不是真正的语义 RAG。

它能演示 SOP 检索，但不足以体现“知识库问答”或“业务处置建议”的深度。

后续可升级为：

- 文档加载
- 分块
- embedding
- vector store
- metadata filter
- 来源引用
- 业务数据 + SOP 联合回答


4. 业务判断模型还不够厚

当前订单、库存、工单、采购之间已经能串起来，但关系仍然偏演示。

例如“订单能不能发货”这个问题，本来可以体现很多制造业业务判断：

- 订单需求量
- 可用库存
- 锁定库存
- 批次状态
- 质检冻结库存
- 采购延期
- 工单补货
- 客户交期
- SOP 处置流程
- 人工确认边界

目前系统已经有这个雏形，但还没有形成明确的业务风险模型。


5. 权限只有工具级，还缺字段级控制

当前权限检查是必要且正确的，但仍然是工具级。

真实企业场景里，不同角色即使能查同一类数据，也不一定能看同样字段。

例如：

- sales 可以看订单和交期，但未必能看供应商敏感信息。
- purchase 可以看采购和物料，但未必能看完整客户信息。
- warehouse 可以看库存批次，但不能执行调账或出库。
- normal_user 只能看公开 SOP。

后续应补一个 response filtering / field masking 层。


6. 管理边界还可以更严

例如 /api/knowledge/rebuild 这类知识库重建接口，虽然不是业务写操作，但属于管理动作，应加 admin 权限。

这类细节能体现工程严谨性。


三、当前项目适合怎么对外表述

建议这样表述：

这是一个制造业企业内部业务查询 Agent 后端 MVP。
它使用模拟 ERP/MES/WMS 数据，支持订单、库存、工单、采购和 SOP 查询。
系统强调只读工具调用、角色权限、审计日志、人工确认边界和 Docker 化启动。

不建议这样表述：

- 生产级平台
- 已接入真实 ERP/MES/WMS
- 已经企业落地
- 自动审批
- 自动出库
- 自动调账
- 完整智能决策系统

这个项目真正值得强调的点不是“AI 很炫”，而是：

1. 制造业场景选得具体。
2. 业务边界清楚。
3. Agent 工具全部只读。
4. 权限检查发生在工具执行前。
5. 拒绝调用也会记录审计日志。
6. 回答必须说明查了什么数据、调用了什么工具、得出什么结论、下一步建议是什么。
7. 涉及风险动作必须提醒人工确认。


四、后续最值得投入的方向

后续不要优先堆更多接口。

更值得做的是把这个项目从“能跑 demo”升级为“有自己的业务思考”。

我建议后续主线是：

主线一：把 Agent 编排层做厚

目标：

从硬编码路由升级为可解释、可追踪、可扩展的业务 Agent 编排层。

建议新增或重构模块：

- app/services/intent_service.py
- app/services/entity_extractor.py
- app/services/tool_planner.py
- app/services/permission_guard.py
- app/services/execution_trace.py
- app/services/answer_composer.py

一次请求的理想链路：

用户问题
-> 意图识别
-> 实体抽取
-> 生成工具计划
-> 权限预检查
-> 执行只读工具
-> 汇总证据
-> 生成业务回答
-> 写审计日志

这会比现在的 if/else 关键词分支更能体现 Agent 编排能力。


主线二：把“订单能不能发货”做成深场景

不要平均用力做很多浅功能。

建议把“订单能不能发货”做成项目主场景。

这个问题天然能串起：

- 销售订单
- 订单明细
- SKU 库存
- 批次
- 锁定库存
- 质检冻结
- 采购到货
- 工单补货
- 客户交期
- SOP
- 人工确认

可以形成一个自己的业务风险模型：

发货可行性 =
订单需求
- 可用库存
- 已锁库存
- 冻结库存
+ 预计采购到货
+ 预计生产完工

风险等级可设计为：

- can_ship
- partial_ship
- shortage_risk
- delivery_delay_risk
- manual_review_required

这类业务模型比单纯增加 CRUD 或接口数量更有含金量。


主线三：把 RAG 从“搜 SOP”升级成“业务处置建议”

当前 SOP 检索能用，但还像资料搜索。

后续应让 RAG 和业务数据结合。

例如用户问：

注塑件外观不良应该怎么处理？

更理想的回答不是只返回 SOP 片段，而是：

- 当前识别为质量异常问题。
- 应隔离相关批次。
- 通知质检复判。
- 记录批次、工单、异常数量。
- 判断返工、报废或让步接收。
- 如果影响客户订单，应同步销售和生产主管。
- Agent 只提供建议，不执行调账、报废、出库或审批。
- 引用 SOP 来源。

这样 RAG 才真正进入业务流。


主线四：把权限从“能不能调用工具”升级为“能看到什么”

当前权限已经是一个好基础。

后续可以增加字段级过滤：

- admin 看完整信息。
- sales 看订单、库存是否满足、交期风险，但隐藏供应商敏感字段。
- purchase 看采购和物料影响，但隐藏客户敏感字段。
- warehouse 看库存、批次、库位，但不看客户商业信息。
- normal_user 只看 SOP。

这个方向很适合体现企业内部系统意识。


五、建议下一阶段优先级

P0.5：先补可信度

1. 给 /api/knowledge/rebuild 加 admin 权限。
2. 增加 docs/agent-orchestration.md，解释 Agent 编排设计。
3. 增加未知订单、未知 SKU、权限不足、空 SOP 的测试。
4. 在 README 增加“当前限制”章节。
5. 修正或避免测试/终端输出里的中文乱码问题。

P1：重构 Agent 编排层

1. 增加 intent/entity/tool-plan 数据结构。
2. 替换 agent_service.py 里的硬编码分支。
3. 支持从自然语言抽取任意订单号、SKU、工单号、采购单号。
4. 生成可返回的 execution_trace。
5. 缺少必要实体时返回澄清问题。

P2：深化真实业务场景

1. 深化“订单能不能发货”。
2. 增加批次状态、冻结库存、BOM、工单补货关系。
3. 增加发货风险等级。
4. 增加角色字段过滤。
5. 增加更多制造业样例数据。

P3：升级 RAG

1. embedding + vector store。
2. SOP metadata。
3. 来源引用。
4. SOP 与业务数据联合生成处置建议。
5. 按角色过滤知识内容。


六、我的总体判断

这个项目方向是值得继续做的。

它的问题不是方向错，而是现在还停留在“P0 demo 能跑通”的阶段。

如果后续只是继续加接口，项目会变成普通 CRUD + 简单问答，亮点不够。

更好的路线是：

少堆功能，多做判断。
少写泛泛的 AI 包装，多做制造业真实链路。
少强调模型能力，多强调工具边界、权限、审计、业务证据和人工确认。

这个项目最有潜力的地方，是把“制造业内部业务查询”做成一个可解释、可审计、只读安全的 Agent 编排系统。

只要后续把 Agent 编排层、业务风险模型、字段级权限和 SOP 处置建议做扎实，它就会从一个半天能搭出来的 MVP，逐步变成一个能体现思考深度的作品。
