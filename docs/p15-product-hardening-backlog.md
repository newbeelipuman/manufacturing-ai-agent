# P15 产品细化与全备功能清单

本文记录 P8-P14 云端 MVP 之后的产品化补强点。当前项目仍是 MVP 原型，使用模拟 ERP/MES/WMS 数据；Agent 工具全部只读，不执行出库、调账、审批、下单等业务写操作。

## 权限中心

当前已实现：

- 权限矩阵按菜单树展示角色可见、可调用、可申请的权限。
- 普通用户可提交平台访问权限申请，并填写申请原因。
- 管理员可审批权限申请，审批意见会写入申请记录。
- 权限申请和审批会写入工具调用日志，工具名分别为 `platform_permission_request` 和 `platform_permission_approval`。
- 前端管理员角色编辑目前是草稿预览，不直接写入后端权限表。

下一阶段需要补齐：

- 新增权限变更审计表，例如 `permission_change_log`。
- 区分权限变动来源：
  - `request_approval`：用户申请，管理员审批。
  - `admin_direct_change`：管理员直接更改角色或用户权限。
  - `system_seed`：系统初始化或演示数据种子。
- 管理员直接更改权限时必须填写备注，记录操作人、目标角色或用户、权限差异、变更前后值、备注、时间和 request id。
- 后端新增只读查询接口，供权限中心展示权限变动历史。
- 后端新增管理员保存接口时，必须进行 admin 权限校验，并写审计日志。

2026-06-25 用户反馈补充：

- 权限中心里管理员编辑角色权限时，必须有“变更备注”输入框；没有用户申请单时，来源默认应记录为“管理员更改权限”。
- 权限变动是高敏操作，不能只改前端状态；后台必须能明确记录来源是“申请审批”还是“管理员更改”。
- 权限矩阵仍应以业务后台常见形式呈现：纵向是菜单树，横向是权限项或权限动作，最多支持三级菜单。
- 角色权限需要区分“当前角色已拥有”“对该角色可见”“普通用户可申请”，不要把所有权限混成一列。

## 审批中心

当前已实现：

- 管理员查看待审批的平台权限申请。
- 审批意见前端必填。
- 通过/驳回后刷新待审批列表。

下一阶段需要补齐：

- 审批列表展示更完整字段：申请时间、申请角色、目标权限中文名、审批来源、审批备注、审批时间。
- 审批详情页展示完整时间线。
- 支持查看已通过、已驳回和全部申请。
- 审批通过后给用户明确提示“刷新权限或重新登录后生效”。

2026-06-25 用户反馈补充：

- 审批中心不能只有“通过/驳回”按钮，必须填写审批原因。
- 审批理由输入框要像正式业务系统，不是临时小输入框；审批理由要落库并可在审计中追溯。
- 审批卡片需要解释申请人为什么申请、管理员为什么通过或驳回、审批后影响哪些权限。

## 页面刷新与草稿保护

当前已实现：

- 运营看板、审计日志、审批中心和部署状态进入页面时自动加载。
- 权限申请原因、审批意见、管理员权限编辑备注会暂存在浏览器本地。
- 管理员编辑角色权限草稿时，刷新或关闭页面会提示确认。

下一阶段需要补齐：

- 对所有有编辑输入的页面统一使用草稿保存机制。
- 提交成功后清理对应草稿，避免旧内容误提交。
- 页面切换时如存在未保存草稿，提示用户确认。
- 网络失败时保留输入内容并显示可重试操作。

2026-06-25 用户反馈补充：

- 审计、运营看板、审批中心、部署状态等查看型页面，切换进页面时应自然刷新，不能让用户每次猜要不要点刷新。
- 有编辑输入的地方必须考虑临时保存，避免网络失败、误点菜单、刷新页面导致用户重输。
- 后续做统一的 `useDraft` 或类似前端机制，按页面 key 保存草稿，提交成功后清理。

## 审计日志

当前已实现：

- Agent 调用日志和工具调用日志列表。
- Agent 日志详情包含决策记录、执行轨迹和工具调用。

下一阶段需要补齐：

- 日志列表进入页面自动刷新后，保留用户当前选择。
- 工具日志也支持详情面板，而不是只展示列表行。
- 权限变动审计、登录审计、部署操作审计单独归类。
- 增加筛选条件：用户、角色、工具名、权限是否允许、时间范围、request id。

2026-06-25 用户反馈补充：

- 审计界面要能看清权限变动来自申请审批还是管理员直接更改。
- 审计日志要能查看详情，不应只停留在列表和路径。
- 权限变动、审批意见、管理员备注、来源、操作者和时间都应进入可查询审计。

## 部署状态

当前已实现：

- `/health` 健康检查。
- 展示云部署报告、演示报告路径。
- 页面说明当前不是文件浏览器，容器日志仍需服务器命令查看。

下一阶段需要补齐：

- 新增只读部署状态 API，返回容器状态、镜像版本、启动时间和最近健康检查结果。
- 新增只读日志 API，按服务名读取最近 N 行日志，管理员可在页面查看 backend/nginx/postgres 日志。
- 增加部署报告下载或预览能力。

2026-06-25 用户反馈补充：

- “部署状态”菜单的作用需要更明确：它应该是运维只读控制台，不只是告诉管理员有哪些文件路径。
- 管理员至少应能在页面看到后端、Nginx、PostgreSQL 等服务的最近日志摘要或详情。
- 页面需要解释哪些信息来自实时 API，哪些信息来自部署报告，避免看起来像一个不能用的文件列表。

## 业务问答

当前已实现：

- 5 个 P0 演示问题。
- 业务结论、建议下一步、检查数据、风险因素、人工复核原因结构化展示。
- 原始审计文本折叠展示。
- 缺少业务编号时返回中文追问。

下一阶段需要补齐：

- 历史会话侧边栏，类似 Codex/ChatGPT 的聊天历史。
- 常见问题入口，例如订单发货、工单开工、采购延期、库存批次、异常 SOP。
- 会话标题自动生成。
- 允许继续追问，并把上一轮实体作为上下文。
- 普通用户视角进一步弱化英文 code，仅在审计折叠区保留。

2026-06-25 用户反馈补充：

- 普通工人或没有英语水平的管理者看不懂英文 intent、risk、tool code 和 key-value 证据，业务问答默认必须用中文业务语言。
- 英文 code 可以保留在“原始回答与审计文本”等折叠区，不能作为主反馈。
- 常见问题、历史聊天、继续追问是后续项目全备功能的一部分，需要先记录，后续逐步实现。
